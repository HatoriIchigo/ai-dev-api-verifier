# CLAUDE.md

このリポジトリで作業する AI エージェント（Claude Code 等）向けの作業原則とプロジェクト情報です。

## 応答原則

- **日本語で応答すること。** コミットメッセージ・コードコメント・ドキュメントも原則として日本語で記述する（コード識別子や標準的な技術用語は英語で可）。
- 推測で答えず、コードや設定ファイルを確認してから回答する。特に redocly のルールやバージョン依存の挙動は実機（`redocly lint` 等）で確認する。
- 影響の大きい設計判断（ルールの追加/削除、検証ロジックの二重化など）は、独断で進めず方針を確認する。
- 作業後は必ず `./tests/run_tests.sh` を実行し、全テストが通ることを確認してから完了を報告する。

## 商用コードとしての原則

このプロジェクトは商用リリースで利用される前提。以下を必ず守る。

### シェルスクリプト

- スクリプト冒頭で `set -euo pipefail` を維持する（未定義変数・パイプ失敗を握りつぶさない）。
- 変数展開は必ずクォートする（`"$var"`）。コマンド置換も同様。
- 入力は必ず検証する（引数の有無・ファイルの存在チェックなど）。異常時は **stderr にメッセージを出し、非ゼロで終了**する。
- 失敗を握りつぶさない。各検証の成否を集約し、**1つでも失敗したら終了コード 1** を返す（CI ゲートとして機能させる）。
- 終了コードは契約。成功=0 / 失敗=1 を厳守し、失敗時は原因の内訳を出力する。
- 外部バイナリのダウンロードは一時ファイル経由で配置する（部分書き込み・"Text file busy" の回避）。
- 破壊的・外部影響のある操作（グローバルインストール、ファイル削除）は影響範囲を明示する。

### 設定（redocly.yaml）

- **ルールを緩める変更は必ず理由をコメントで残す**（いつ・なぜ・誰の合意か）。レビュー可能性を最優先する。
- ベースラインは `recommended`。安易に `minimal` へ下げない（warning 化は品質ゲートの空洞化を招く）。
- ルールを無効化（`off`）する場合は、それが妥当な業務上の理由（例: 社内 IF 仕様でライセンス概念が無い）に基づくこと。

### 検証ロジックの設計方針

- **redocly で表現できる検証は redocly に寄せる**（自前実装と二重化しない）。
  例: `operationId` 必須・`example` 必須は redocly のルールで担保する。
- **redocly で表現できない独自規約のみ自前チェック（yq/bash）で実装する**。
  例: `x-internal` の値・行末コメント規約、先頭フロントマター。
- 検証を追加・変更したら、それを **実際に発火させるテスト**を必ず追加する（ルールを有効化しただけで未検証の状態を残さない）。

### テスト

- 各テストは **1つの失敗要因だけ**を発生させるよう設計する（`redocly` / `operations` / `frontmatter` を分離）。
- ランナーは終了コードだけでなく **どの検証で失敗したか**まで照合する。この分離を壊さないこと。
- ルールやロジックを追加したら、正常系（pass）と異常系の両方のケースを用意する。
- 暗黙の打ち切り（カバレッジの穴）を残さない。テスト対象外がある場合は明示する。

## プロジェクト概要

OpenAPI 仕様書を 3 段階で検証する（詳細は `README.md`）。

1. **redocly lint** — OpenAPI 標準 + 商用向けルール（`redocly.yaml`）
2. **x-internal チェック**（`verify.sh` の `check_operations`）— 存在・true/false・true 時のコメント規約
3. **フロントマターチェック**（`verify.sh` の `check_frontmatter`）— 先頭コメントの必須項目

## 主要コマンド

```bash
# 検証の実行
./verify.sh <api-spec-file>

# 全テストを並列実行（デフォルト 4 並列）
./tests/run_tests.sh
./tests/run_tests.sh 8          # 並列数を指定
MAX_JOBS=2 ./tests/run_tests.sh

# 構文チェック
bash -n verify.sh

# 個別 spec の lint 内訳を確認
redocly lint tests/test_01/openapi.yaml
```

## 重要な設計判断（変更時は要確認）

- `operationId` の必須チェックは **redocly に一本化**済み（自前チェックには戻さない）。
- `example` 必須は redocly（`scalar-property-missing-example`）に任せる方針。
- `x-internal` の値・コメント規約、フロントマターは redocly で表現できないため **自前チェックで実装**している。
- 5xx レスポンスを検証する redocly ルールは存在しない（要件化する場合は自前実装が必要）。

## ファイル構成

- `verify.sh` — 検証スクリプト本体
- `redocly.yaml` — redocly のルール設定（変更理由はコメントで管理）
- `tests/run_tests.sh` — 並列テストランナー
- `tests/test_NN/` — `openapi.yaml`（検証対象）と `expected`（期待結果）

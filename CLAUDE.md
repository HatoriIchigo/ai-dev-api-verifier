# CLAUDE.md

このリポジトリで作業する AI エージェント（Claude Code 等）向けの**全ツール共通**の作業原則です。
本リポジトリは複数の検証ツールを収録するモノレポです。**各ツールの詳細仕様は `docs/` 配下に分離**して
管理し、本ファイルは肥大化させず共通原則とツールへの入口だけを保ちます。

## 収録ツールと詳細ドキュメント

| ツール | 概要 | 詳細 |
|--------|------|------|
| `ifspec-verifier/` | OpenAPI 仕様書を redocly + 独自規約で検証 | [docs/ifspec-verifier.md](docs/ifspec-verifier.md) |
| `layered-checker/` | 設計 JSON を構成ルール・レイヤールールで検証 | [docs/layered-checker.md](docs/layered-checker.md) |
| `java-builder/` | Java ソースを AST 解析して構成・規約を検証 | [docs/java-builder.md](docs/java-builder.md) ／ [java-builder/CLAUDE.md](java-builder/CLAUDE.md) |
| `stack-selector/` | 技術スタック決定 JSON のキー構造を検証 | [docs/stack-selector.md](docs/stack-selector.md) ／ [stack-selector/CLAUDE.md](stack-selector/CLAUDE.md) |

`layered-checker` と `java-builder` が検証するレイヤー・repository などの**構成ルール本体は
[docs/code-rule.md](docs/code-rule.md) に一元管理**する（両ツール共通の正本）。ルールの追加・変更は
まず `docs/code-rule.md` に反映し、各ツールの実装・テストを揃える。

- 特定のツールに手を入れる前に、まず該当ツールの詳細ドキュメントを読むこと。
- ツール固有の設計判断・コマンド・テスト方針は各 docs に記載する。**本ファイルへ固有情報を増やさない**
  （肥大化を避けるため、ツール固有の記述は `docs/<tool>.md` へ退避する）。

## 応答原則（全ツール共通）

- **日本語で応答すること。** コミットメッセージ・コードコメント・ドキュメントも原則として日本語で記述する（コード識別子や標準的な技術用語は英語で可）。
- 推測で答えず、コードや設定ファイルを確認してから回答する。特にツール・バージョン依存の挙動は実機（`redocly lint`・`jq`・Docker ビルド等）で確認する。
- 影響の大きい設計判断（ルールの追加/削除、検証ロジックの二重化、ディレクトリ構成の変更など）は、独断で進めず方針を確認する。
- 作業後は必ず**該当ツールの検証/テストを実行**し、通ることを確認してから完了を報告する（例: `ifspec-verifier/tests/run_tests.sh`、`layered-checker/tests/run_tests.sh`、各 `verify.sh`、java-builder の Docker ビルド）。

## 商用コードとしての原則（全ツール共通）

このプロジェクトは商用リリースで利用される前提。以下を必ず守る。

### シェルスクリプト

- スクリプト冒頭で `set -euo pipefail` を維持する（未定義変数・パイプ失敗を握りつぶさない）。
- 変数展開は必ずクォートする（`"$var"`）。コマンド置換も同様。
- 入力は必ず検証する（引数の有無・ファイルの存在チェックなど）。異常時は **stderr にメッセージを出し、非ゼロで終了**する。
- 失敗を握りつぶさない。各検証の成否を集約し、**1つでも失敗したら終了コード 1** を返す（CI ゲートとして機能させる）。
- 終了コードは契約。成功=0 / 失敗=非0 を厳守し、失敗時は原因の内訳を出力する。
- 外部バイナリ・パッケージの導入は影響範囲を明示し、ダウンロードは一時ファイル経由で配置する（部分書き込み・"Text file busy" の回避）。
- 破壊的・外部影響のある操作（グローバルインストール、ファイル削除）は影響範囲を明示する。

### 設定・ルール

- **ルールを緩める変更は必ず理由をコメントで残す**（いつ・なぜ・誰の合意か）。レビュー可能性を最優先する。
- 品質ゲートを安易に空洞化しない（例: redocly のベースラインを安易に `minimal` へ下げない／warning 化しない）。
- ルールを無効化（`off`）する場合は、妥当な業務上の理由（例: 社内 IF 仕様でライセンス概念が無い）に基づくこと。

### 検証ロジックの設計方針

- **既存ツールで表現できる検証はそのツールに寄せる**（自前実装と二重化しない）。
  例: ifspec-verifier では `operationId` 必須・`example` 必須を redocly のルールで担保する。
- **既存ツールで表現できない独自規約のみ自前チェック（bash/yq/jq、AST 等）で実装する**。
- 検証を追加・変更したら、それを **実際に発火させるテスト**を必ず追加する（ルールを有効化しただけで未検証の状態を残さない）。
- 各テストは **1つの失敗要因だけ**を発生させるよう設計し、ランナーはどの検証で失敗したかまで照合する。この分離を壊さないこと。
- 正常系（pass）と異常系の両方のケースを用意し、暗黙の打ち切り（カバレッジの穴）を残さない。対象外がある場合は明示する。

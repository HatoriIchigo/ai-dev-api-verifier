# stack-selector

## 概要

「決定された技術スタック」を表す JSON ファイルの**キー構造**を検証するスクリプト。
値の内容（言語の種別・バージョン書式・ライブラリ名など）は問わず、**キーがそろっているか**だけを
検証する。キー検証は「不足キー」「未知キー」の双方をエラーとする（キー集合の完全一致）。

## 技術スタック

- bash
- jq

## kind ディスパッチ（プロジェクト種別）

トップレベル `kind` で適用スキーマを分岐する。`kind` キーは**必須**（有無のみ検証・値自体は検証しない）。

- `kind` が無い → エラー（必須キー不足）
- `kind == "backend"` → backend 用スキーマで検証（下記）＋構成妥当性チェック
- それ以外（`frontend` / `iac` / 未知の値）→ **未対応として検証をスキップ（pass）**

> 将来 `frontend` / `iac` を追加する余地として、`main` の `case "$kind"` に分岐を足し、
> 専用の `verify_keys_<kind>` を実装するだけで対応できる構造にしてある。

## 入力スキーマ（kind=backend。キーのみ。値の型・内容は検証対象外）

```json
{
  "kind": "backend",
  "language": { "kind": "...", "version": "..." },
  "package":  "...",
  "build":    "...",
  "libraries": [ { "name": "...", "version": "..." } ]
}
```

- トップレベル必須キー: `kind` / `language` / `package` / `build` / `libraries`
- `language` 必須キー: `kind` / `version`
- `libraries` は配列。各要素は `name` / `version` を持つオブジェクト（空配列は許可）
- 上記以外のキー（未知キー）が現れた場合もエラー

## 構成妥当性チェック（助言・終了コードに影響しない）

キー検証に加え、`language.kind` と `package` / `build` の組み合わせが既知の妥当な構成かを
**助言として**確認する（`package`・`build` の双方に適用）。

| language.kind | 許可ツール（package / build） |
| -- | -- |
| `java` | `maven` / `gradle` / `ant` |
| `python` | `pip` |
| `typescript` | `npm` |
| `go` | `go` / `gomod` |
| `rust` | `cargo` |

- **組み合わせは強制しない**: 非許可ツール（例: python + `uv`）でも、未知言語
  （マトリクスに無い `kind`）でも**エラーにはしない**（終了コードはキー検証のみが左右する）。
- 出力は INFO のみ:
  - 一致 → 「構成は妥当です（既知の構成）」
  - 既知言語で非許可ツール → 「既知の妥当な組み合わせに一致しません（許容・非エラー）」
  - 未知言語 → 「構成妥当性チェックは対象外です」

> CI ゲートとして組み合わせも失敗扱いにしたい場合は、`--strict` 相当の拡張で
> MISMATCH を `exit 1` 化できる（現状は未実装。要望に応じて追加）。

## 使い方

```sh
./verify.sh <input.json>
```

終了コード: `0`=成功 / `1`=失敗（不足キー・未知キー・構造不正・入力エラー）。
失敗時は不足／未知キーの内訳を stderr に出力する。構成妥当性は助言のため終了コードに影響しない。

## テスト

```sh
./tests/run_tests.sh
```

`tests/test_*/` に `input.json` と `expected`（`pass`/`fail`）を置き、必要に応じて `match`
（出力に含むべき部分文字列。1行=1つ）で失敗要因を固定する。各異常系テストは
**1つの失敗要因だけ**を発生させる。
